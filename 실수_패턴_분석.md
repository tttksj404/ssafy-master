## 1. Visited 체크 실수 (가장 빈번)

### 1-A. 빈 셀/중립 셀 처리 누락
```python
# 3184bfs정석여러변수.py — 첫 시도에서 '.' 셀을 무시함
if grid[nr][nc] == 'o':
    visited[nr][nc] = True
    queue.append(...)
# ← '.' 셀에 대한 처리가 없어서 BFS가 중간에 끊김
```
**교훈**: BFS 조건문 작성 시 벽이 아닌 모든 셀 타입(빈칸, 시작점, 목적지 등)을 빠짐없이 처리해야 함.

### 1-B. visited를 리스트로 사용 → O(N) 탐색
```python
# 1219 DFS — visited=[] + if current not in visited
visited = []                    # ← O(N) lookup
visited.append(current)
if current not in visited:      # 매번 전체 순회
```
**수정**: `visited = [False] * N` 또는 `visited = set()` 사용. `01_자료구조/21464`에서 본인이 직접 list→set 효율화를 경험했는데, DFS 쪽에서는 아직 적용 안 된 파일이 있음.

### 1-C. 글로벌 큐 재사용 (위험한 패턴)
```python
# 2468, 2667, 1926, 3184 등 다수
queue = deque()  # ← 모듈 레벨 전역 변수

def bfs(x, y):
    queue.append((x, y))  # 이전 BFS에서 안 빠진 데이터가 남아있을 수 있음
```
**교훈**: BFS 함수 안에서 `queue = deque()` 로 매번 새로 만들기. 전역 큐는 잔여 데이터로 오답 발생 가능.

## 2. R/C (행/열) 혼동 (치명적)

### 2-A. 입력 파싱에서 W와 H 뒤바뀜
```python
# 5656벽돌깨기.py
W, H = map(int, input().split())
brick_map = [list(map(int, input().split())) for _ in range(W)]
#                                                         ^^^ H여야 함
```
**교훈**: 입력받을 때 `for _ in range(행의 수)` 인지 반드시 확인. W(폭)=열, H(높이)=행.

### 2-B. 배열 선언과 접근 순서 불일치
```python
visited = [[0]*C for _ in range(R)]  # R행 C열
visited[c][r]  # ← 거꾸로 접근 → IndexError (R≠C일 때)
```
**규칙**: 선언: `[_ for _ in range(R)]` → 접근: `[i][j]` 에서 i=행(R), j=열(C). 항상 R→C 순서 통일.

### 4-B. 경계 조건에 데드코드
```python
# 2105 디저트카페
for i in range(N):
    for j in range(N):
        if j==0 or i==N or j==N or i==N-1:  # i==N, j==N은 절대 True가 안 됨
            continue                          # range(N)이면 최대 N-1
```
**수정**: `i==N` → `i==N-1`, `j==N` → `j==N-1`.

## 11. 최신 추가: `bfs.py` (K번 최댓값 이동)에서 드러난 패턴

### 11-A. 전역 큐 재사용 패턴 재등장
```python
# bfs.py
q = deque()                 # 전역

def bfs(r, c):
    q.append((r, c))
    while q:
        r, c = q.popleft()
```
**포인트**: 현재 코드는 `while q`를 끝까지 돌아서 우연히 안전하지만, 예외/조기 `return`이 생기면 다음 BFS 호출에 큐 찌꺼기가 남을 수 있음.  
**수정 습관**: `q`는 함수 안에서 매번 새로 만들기 (`q = deque([(r, c)])`).

### 11-B. 델타 배열을 루프 안에서 매번 재생성
```python
for a in range(4):
    dr = [-1, 1, 0, 0]
    dc = [0, 0, -1, 1]
    nr = r + dr[a]
    nc = c + dc[a]
```
**포인트**: 정답에는 영향이 없지만, 반복 생성으로 가독성이 떨어지고 인덱스 실수 가능성이 커짐.  
**수정 습관**: `DIRS = [(-1,0), (1,0), (0,-1), (0,1)]`를 루프 밖 상수로 두고 `for dr, dc in DIRS`.

### 11-C. 동점 우선순위는 "튜플 정렬 한 번"으로 고정
```python
candidates.append((-grid[nr][nc], nr, nc))
candidates.sort()
best = candidates[0]
```
**포인트**: 값은 내림차순(음수), 행/열은 오름차순을 동시에 만족하는 안전한 패턴.  
**교훈**: 이런 우선순위 문제는 if 분기 여러 겹보다 튜플 정렬 규칙으로 고정하면 실수율이 줄어듦.

### 11-D. 종료 조건 분리 체크는 유지할 것
```python
if not candidates:
    return None

for _ in range(k):
    result = bfs(r, c)
    if result is None:
        break
```
**포인트**: "K번 전에 이동 불가면 즉시 종료"라는 문제 조건을 정확히 반영함.  
**교훈**: 반복 횟수(`k`)와 실제 진행 가능 여부(`result is None`)를 반드시 분리해서 판단.

## 12. 최신 추가: `bfs2.py` (돌 잘 치우기)에서 나온 실수 총정리

### 12-A. "결정"과 "탐색"을 섞으면 꼬인다
```python
# 잘못된 접근
# BFS 도중에 돌을 만나면 remove_stones를 올리며 진행
q.append((r, c, remove_stones, count))
```
**문제점**: 어떤 돌을 치울지(조합 결정)를 탐색 중에 처리해서, 시작점이 여러 개일 때 `M` 제약을 일관되게 공유하기 어려움.  
**정석**: `(돌 M개 선택)`을 먼저 하고, 그 상태에서 BFS로 도달 칸 수를 측정.

### 12-B. 시작점이 여러 개인데 BFS를 따로 돌리면 중복 카운트
```python
# 잘못된 접근
for s in starts:
    total += bfs(s)
```
**문제점**: 시작점 A/B가 공통으로 도달하는 칸이 중복 집계됨.  
**정석**: `q = deque(start_points)`로 한 번에 시작하는 멀티 소스 BFS.

### 12-C. deque 초기화 방식 혼동
```python
q.append(start_points)  # 리스트 통째로 1개 원소
q = deque(start_points) # 좌표 원소들이 각각 큐에 들어감
```
**문제점**: 큐 원소 타입이 `(list)`와 `(tuple)`로 섞여 언패킹/순회 로직이 깨짐.  
**정석**: 멀티 소스 시작은 `deque(start_points)` 또는 반복 `append((r,c))`로 통일.

### 12-D. 변수 쉐도잉(가장 많이 터지는 한 끗 오타)
```python
curr_r, curr_c = q.popleft()
nr = r + dr[a]   # <- curr_r가 아니라 r를 써서 오답
nc = c + dc[a]
```
**문제점**: 바깥 루프의 `r, c`(마지막 시작점 값)를 계속 사용해 BFS가 퍼지지 않음.  
**정석**: `curr_`, `next_`, `start_`처럼 역할 기반 변수명으로 구분.

### 12-E. 상태 복구 원칙은 아주 잘한 부분
```python
for r, c in selected_stones:
    grid[r][c] = 0
...
for r, c in selected_stones:
    grid[r][c] = 1
```
**의미**: 조합/백트래킹에서 다음 케이스의 독립성을 지키는 핵심 습관.  
**유지할 점**: "내가 바꾼 전역 상태는 내가 복구하고 나온다"를 계속 유지.

### 12-F. 계산량 불안은 식으로 판단
```text
총 복잡도 ≈ C(돌개수, M) * O(N^2)
```
**교훈**: `N=100`만 보고 겁먹기보다, 실제 병목은 `조합 수`라는 점을 먼저 계산해서 판단.

### 12-G. 이번 문제에서 고정할 구현 템플릿
```python
stones = [...]  # 1인 칸 좌표
for selected in combinations(stones, m):
    remove(selected)
    run_multi_source_bfs()
    restore(selected)
```
**교훈**: "조합 + 멀티 소스 BFS + 원복"은 유사 유형에서 재사용 가능한 정답 패턴.
